# STS2 Mod 项目工作规则

本仓库是《Slay the Spire 2》mod 研究与开发工作区，不是游戏安装目录。

## 项目定位

- 当前项目重心是：研究、验证、梳理 STS2 的本地 mod 安装与加载链路。
- 当前主线聚焦：
  - manifest 实际位置与最低要求
  - mod 安装位置
  - 加载 gate / warning 同意状态
  - SmokeMod 最小闭环验证
- 不要无边界发散到大规模功能开发、未经验证的 API 封装或复杂框架设计。

## 工作区与游戏目录边界

- 真实游戏目录位于仓库外。
- 游戏根路径以 `local/game-path.txt` 为准。
- 不允许把 Steam 游戏安装结构直接并入仓库。
- 不允许把源码、实验文件、临时脚本直接写进游戏安装目录。
- 游戏目录只用于：
  - 读取参考文件
  - 安装最终成品
  - 运行验证

## 与游戏目录交互规则

- 对游戏目录的读写操作优先通过 `shared/scripts/` 中的脚本完成。
- 如果需要分析游戏文件，应先复制到 `references/`，再在工作区中分析。
- 不要手工把整个 mod 目录、源码目录或文档目录复制进游戏安装区。

## 事实与推测

- 不要把推测写成事实。
- 不要编造 STS2 官方 API、类名、方法名、manifest schema 或加载规则。
- 未验证的路径、入口点、字段结构、加载条件，必须明确标记为“高可能”或“待确认”。
- 结论优先写入高密度结论文档，不要只留在聊天记录里。

## 新线程默认阅读顺序

新线程默认先读以下文件，不要先通读整个 `docs/`：

1. `AGENTS.md`
2. `docs/current-status.md`
3. `docs/next-task.md`
4. `docs/thread-handoff.md`
5. `docs/decisions.md`

只有当这些文件不足以支撑当前任务时，才按需下钻：

- `docs/index.md`
- `docs/findings.md`
- 其他专题记录

## 历史日志使用原则

- 历史日志和旧研究记录默认不是新线程必读内容。
- 新线程应优先读结论文档，而不是重新扫描旧日志。
- 只有在 `docs/current-status.md` 与 `docs/next-task.md` 无法回答当前问题时，才去检索旧日志、旧实验记录或原始证据文件。

## 目录职责

- `mods/`：各个 mod 的独立开发目录。
- `shared/`：共享脚本、模板和通用说明。
- `references/`：从游戏目录复制出的参考文件与分析笔记。
- `local/`：仅与当前机器有关的路径和安装说明。
- `docs/`：结论、导航、决策、任务与专题记录。
- `docs/archive/`：不再属于接班必读的旧记录或归档资料。

## 编辑原则

- 优先维护“工作区”和“游戏安装区”的清晰边界。
- 新增脚本默认保持保守，不做危险覆盖。
- 新线程接班时，优先更新 `docs/current-status.md`、`docs/next-task.md`、`docs/thread-handoff.md`，避免继续依赖聊天历史恢复上下文。
